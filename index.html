What Viam is

Viam is not one tool. It is three layers that work together: a RAG knowledge base that never forgets a decision, a simulated studio of role-specific agents that plan and build like a real team, and a quantitative-data gate that refuses to treat an idea as validated until it has actual numbers behind it. Most AI game tools are fast at one thing: generating a document, writing a script, producing an asset. Viam's job is different. It makes sure the thing being generated deserved to exist in the first place.

What's inside

RAG knowledge base

Three collections, one for programming best practices, one for game design documents and design decisions, one for production methodology. Every past decision is logged with its business rationale, so the system can answer "what did we decide, and why" instead of starting from zero every time.

Multi-agent studio

Instead of one model doing everything, specialized agents split the work the way a real studio would: a producer, a game designer, an art director, a marketing lead, each grounded in the same knowledge base and routed to by an orchestrator.

Quantitative-data gate

Before a game design document is treated as validated, it has to contain concrete numbers: market size, competitor benchmarks, target retention, CPI, ARPU. Enthusiasm alone does not pass.

Read-only dev tool access

External collaborators get scoped, read-only access to the knowledge base and a reviewed way to submit their own decisions back, without needing write access to the core system.

viam CLI (Phase 1, new)

The newest layer: a standalone terminal agent, built on Anthropic's Claude Agent SDK, that carries the quantitative-data gate out of this system and into a general-purpose tool anyone can run against their own project.

Where the CLI came from

We asked a simple question: why not build a CLI like Pi or Hermes Agent, but for game development? So we looked closely at what already exists: Pi's minimal agent harness, Hermes Agent's self-improving memory, Bezi and Coplay's deep Unity integration, Unity's own native AI tools, multi-agent asset pipelines like Atlas AI Studio, and GDD generators like Blueprint, StraySpark, and Ludo.ai.

Every one of them was already good, often very good, at generation and execution speed. None of them asked whether the idea underneath deserved to be built in the first place. That gap is where the CLI layer starts, on top of a discipline the rest of the system already had.

What we didn't build

  • Our own agent loop. The Claude Agent SDK already gives us tool execution, context management, and MCP support. Pi and Hermes Agent already solved this well as general-purpose harnesses, so re-solving it would have added nothing.
  • A Unity execution layer. Open-source Unity MCP servers and Coplay already handle safe in-editor mutation properly. That problem is solved, we're not re-solving it.
  • An economy or balance simulator. Tools like Itembase and Leap are already better at this than anything we'd ship on day one.
  • A full IDE. Viam is a discipline layer today, not a replacement for your editor.

Roadmap

1

Working CLI for our own projects (now)

Prove the discipline layer holds up against real games before anyone else touches it.

2

Integration inside a partner platform

Rather than selling Viam standalone, run it as a layer inside a partner's existing product surface. Under discussion, not finalized. The core stays independent of any single partner so it isn't stranded if that doesn't happen.

3

Full-scale product (conditional, distant)

Only after Phase 1 and 2 earn real validation. Not a near-term plan, and no architecture decisions are being made around it yet.
