What Viam is

Viam is not one tool. It is three layers that work together: a RAG knowledge base that never forgets a decision, a simulated studio of role-specific agents that plan and build like a real team, and a quantitative-data gate that refuses to treat an idea as validated until it has actual numbers behind it. Most AI game tools are fast at one thing: generating a document, writing a script, producing an asset. Viam's job is different. It makes sure the thing being generated deserved to exist in the first place.

What's inside

RAG knowledge base

Three separate collections, kept separate on purpose so a query never mixes unrelated context. One holds programming best practices (Clean Code and free programming books, over a hundred thousand indexed chunks). One holds game design documents and logged design decisions, each decision tagged with the business goal it served (retention, monetization, engagement, and so on) and the designer's own stated reasoning, not a paraphrase of it. One holds production methodology: project management, software architecture, infrastructure. Every past decision is queryable, so the system can answer "what did we decide, and why" instead of starting from zero every time.

Multi-agent studio

Instead of one model doing everything, specialized agents split the work the way a real studio would, each with a defined role: a CEO agent for vision and scheduling, a CTO agent for technical maintenance and audits, a game designer for mechanics, an art director for translating visual designs into usable data, a marketing lead for store and ad strategy, a production coordinator that delegates coding work without writing code itself, and an orchestrator that routes every incoming request to the right one. All of them query the same knowledge base before acting, not just their own memory.

Quantitative-data gate

Before a game design document is treated as validated, it is scanned for concrete markers: percentages, dollar amounts, named retention windows (D1, D7, D30), and terms like TAM, SAM, KPI, CPI, ARPU, CAC, LTV, CAGR. A document needs at least two distinct markers to pass. A document that only expresses enthusiasm, no matter how well written, is quarantined instead of accepted.

Read-only dev tool access

External collaborators get scoped, read-only access to the knowledge base, plus a submission path for their own decisions: they write it locally, the tool opens a pull request against the core system automatically, and a human reviews it before it joins the shared knowledge base. No write access to the core system is ever required to do the job.

viam CLI (Phase 1, new)

The newest layer: a standalone terminal agent, built on Anthropic's Claude Agent SDK, that carries the quantitative-data gate out of this system and into a general-purpose tool anyone can run against their own project. Two commands exist today: viam chat, a terminal agent loop that reports which model answered and which tools it called, and viam check-gdd, which runs the same quantitative-data gate described above against any file you point it at.

Where the CLI came from

We asked a simple question: why not build a CLI like Pi or Hermes Agent, but for game development? So before writing a line of code, we looked closely at what already exists.

Pi

An open-source agent harness built around a deliberately minimal core: four tools, a system prompt under a thousand tokens, and everything else added through a TypeScript extension system. Proof that a lean harness can still be fully capable.

Hermes Agent

Nous Research's self-hosted, self-improving agent. It writes its own reusable skills after a complex task, keeps a three-layer memory that survives across sessions, and reaches you over Telegram, Discord, Slack, WhatsApp, or Signal, not just a terminal.

Bezi and Coplay

Two different takes on a Unity-native AI assistant. Bezi indexes your whole project (scripts, scenes, assets, even your GDD) and answers or edits from inside the editor. Coplay goes further: a plugin written directly in C#, living inside the Unity Editor itself, with a system prompt tuned specifically for Unity work.

Unity MCP and Unity AI

The open-source Unity MCP servers expose dozens of structured editor operations (manage scenes, scripts, prefabs, materials) safely, over the Model Context Protocol, to any AI client. Unity's own native AI adds an in-editor assistant plus generators for sprites, textures, and animation, no separate subscription required.

Atlas AI Studio

A funded, multi-agent asset production platform, live on the Google Cloud Marketplace, generating 2D and 3D assets, textures, and environments with fleets of coordinated agents rather than one model.

Blueprint, StraySpark, and Ludo.ai

Three different GDD generators, each turning a rough idea into a structured design document (mechanics, systems, MVP scope, milestones) in minutes, some requiring no account at all.

Every one of them was already good, often very good, at generation and execution speed. None of them asked whether the idea underneath deserved to be built in the first place. That gap is where the CLI layer starts, on top of a discipline the rest of the system already had.

What we didn't build

  • Our own agent loop. The Claude Agent SDK already gives us tool execution, context management, and MCP support. Pi and Hermes Agent already solved this well as general-purpose harnesses, so re-solving it would have added nothing.
  • A Unity execution layer. Open-source Unity MCP servers and Coplay already handle safe in-editor mutation properly. That problem is solved, we're not re-solving it.
  • An economy or balance simulator. Tools like Itembase and Leap are already better at this than anything we'd ship on day one.
  • A full IDE. Viam is a discipline layer today, not a replacement for your editor.

Roadmap

1

Working CLI for our own projects (now)

Prove the discipline layer holds up against real games before anyone else touches it. Concretely: both CLI commands run reliably against our own in-development titles, authentication works without daily friction, and the quantitative-data gate catches a real weak idea at least once before Phase 2 starts.

2

Integration inside a partner platform

Rather than selling Viam standalone, run it as a layer inside a partner's existing product surface. Under discussion, not finalized. The core stays independent of any single partner, built as a portable module rather than something wired directly into their stack, so it isn't stranded if that doesn't happen.

3

Full-scale product (conditional, distant)

Only after Phase 1 and 2 earn real validation, meaning outside studios have actually used it and would pay for it, not just that it works for us. Not a near-term plan, and no architecture decisions are being made around it yet.
