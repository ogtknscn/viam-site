Viam

Validate before you build.

A game-development-specific CLI. Not a faster way to generate a GDD or write Unity code — those problems are already solved well by other tools. Viam is the discipline layer that asks whether your idea has real numbers behind it before you spend months on it.

Phase 1 — private, in active use on our own game projects

What Viam is

Every AI coding agent — general-purpose ones and the growing list of game-specific ones — is good at speed: generating a document faster, writing a script faster, producing an asset faster. None of them stop to ask whether the idea underneath was validated with real numbers before you started treating it as real.

Viam's answer is a quantitative-data gate: before an idea or GDD is treated as ready to build on, it has to show actual numbers — market size, competitor benchmarks, target retention/CPI/ARPU — not just enthusiasm. That check is the product's spine, not a feature bolted on at the end.

Where this came from

We started by asking a simple question: why not build a CLI like Pi or Hermes Agent, but for game development? So we looked closely at what already exists — Pi's minimal agent harness, Hermes Agent's self-improving memory, Bezi and Coplay's deep Unity integration, Unity's own native AI tools, multi-agent asset pipelines like Atlas AI Studio, and GDD generators like Blueprint, StraySpark, and Ludo.ai.

Every one of them was already good — often very good — at generation and execution speed. None of them asked whether the idea underneath deserved to be built in the first place. That gap, not a faster CLI, is where Viam actually starts.

What exists today

viam check-gdd <file>

Scans an idea/GDD document for concrete numbers — market size, KPI benchmarks, target metrics. Passes or fails, no hedging.

viam chat "<prompt>"

A terminal agent harness built on Anthropic's Claude Agent SDK, proving the plumbing works before anything game-specific is layered on top of it.

What we didn't build

  • Our own agent loop — the Claude Agent SDK already gives us tool execution, context management, and MCP support. Pi and Hermes Agent already solved this well as general-purpose harnesses; re-solving it would have added nothing.
  • A Unity execution layer — open-source Unity MCP servers and Coplay already handle safe in-editor mutation properly. That problem is solved; we're not re-solving it.
  • An economy/balance simulator — tools like Itembase and Leap are already better at this than anything we'd ship on day one.
  • A full IDE — Viam is a discipline layer today, not a replacement for your editor.

Roadmap

1

Working CLI for our own projects (now)

Prove the discipline layer holds up against real games before anyone else touches it.

2

Integration inside a partner platform

Rather than selling Viam standalone, run it as a layer inside a partner's existing product surface. Under discussion — not finalized, and the core stays independent of any single partner so it isn't stranded if that doesn't happen.

3

Full-scale product (conditional, distant)

Only after Phase 1 and 2 earn real validation — not a near-term plan, and no architecture decisions are being made around it yet.
